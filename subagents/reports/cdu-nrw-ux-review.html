Accessibility

10 issues · WCAG 2.1 AA
Images Missing Descriptive Alt Text
CriticalAccessibility
News thumbnails — captions are tiny credits, no alt text

Multiple images across all pages (hero carousel, news thumbnails, article cards) have empty alt="". Screen readers announce nothing, leaving blind users with no context about visual content.

alt="" on hero slides, news cards, article preview images sitewide
How to fix
  • Audit every <img> — if it conveys information, write alt text that describes its purpose, not its appearance.
  • For news thumbnails use the article title: alt="CDU NRW Veranstaltung – Neujahrsempfang 2026"
  • Hero carousel: each slide's image should describe the scene, e.g. alt="Hendrik Wüst bei der Wahlkampf-Tour Baden-Württemberg"
  • Purely decorative images (background textures, dividers): keep alt="" and add role="presentation"
  • In Drupal 7: set the alt field in the Media module for each image asset — it propagates everywhere the image is used.
WCAG 1.1.1 Non-text Content
Image CAPTCHA Blocks Blind Users
CriticalAccessibility
Kontakt form — image CAPTCHA with no audio alternative

The contact form uses a visual CAPTCHA with no audio alternative. Blind and low-vision users cannot complete the form — they are entirely locked out from contacting the party.

Contact form: "CAPTCHA image verification" — no audio/text fallback present
How to fix
  • Best option: Replace with Google reCAPTCHA v3 (invisible, no user interaction required). Add the Drupal module recaptcha.
  • Alternative: Use a honeypot field (hidden input that bots fill, humans don't) via Drupal's Honeypot module.
  • If keeping image CAPTCHA: add an audio CAPTCHA button next to it with aria-label="Audio CAPTCHA abspielen"
  • Also add a link "CAPTCHA nicht lesbar? Neu laden" with keyboard access.

Minimal ARIA Implementation Sitewide
HighAccessibility

Interactive components (dropdown menus, carousel, social media wall) lack role, aria-expanded, aria-label, and aria-live attributes. Assistive technologies cannot convey widget state.

No role, aria-label, aria-describedby, or aria-expanded found on reviewed pages
How to fix
  • Navigation dropdowns: add aria-expanded="false" to toggle buttons; update to true on open via JS.
  • Carousel: add role="region" aria-label="Aktuelle Highlights", aria-live="polite" on the active slide container.
  • Active nav item: add aria-current="page" on the current page link.
  • Page landmark: wrap main content in <main>, navigation in <nav aria-label="Hauptnavigation">.
<button aria-expanded="false" aria-controls="menu-partei">Partei</button> <ul id="menu-partei" hidden>...</ul>

Carousel Not Keyboard Accessible
HighAccessibility
Hero carousel (cycle.js) — keyboard/pause controls unverified

The hero carousel (40+ slides, cycle.js) auto-rotates with no pause button. WCAG 2.2.2 requires moving content that lasts more than 5 seconds to have a mechanism to pause, stop, or hide it.

ubg_bean_slideshow with cycle.js — no pause/stop control visible
How to fix
  • Add a visible Pause/Play button: <button aria-label="Slideshow pausieren">⏸</button>
  • Pause on keyboard focus of any slide content automatically.
  • Ensure prev/next arrows are <button> elements (not <a> or <div>) with descriptive aria-label.
  • Add aria-roledescription="slide" to each slide, and aria-label="Slide X von Y".
  • Long-term: replace 40-slide carousel with a static featured section (max 3 items).

<h1>Startseite</h1>

Cookie Banner Blocks All Content on Mobile
MediumAccessibility
Mobile 390px — cookie banner occupies entire viewport

On mobile, the cookie consent banner takes up the entire below-fold area, covering all page content. Keyboard and screen reader users are forced to navigate the banner before any page content is accessible. Focus is not trapped in the banner, which is also an issue.

Mobile screenshot: cookie text + 2 checkboxes + 2 buttons = full viewport below header
How to fix
  • Redesign as a compact bottom bar (max 120px tall) with a "Mehr erfahren" expand option.
  • On open: trap keyboard focus inside the dialog using role="dialog" and aria-modal="true".
  • Ensure "Alle ablehnen" is as visually prominent as "Alle akzeptieren" — both buttons should have equal styling.
  • Return focus to the triggering element (top of page) after consent is given.

Required Fields — Asterisk Only, No Programmatic Association
MediumAccessibility

Contact form required fields are marked with * only. No required attribute or aria-required, so screen readers don't announce them as required.

Vorname*, Nachname*, PLZ*, E-Mail*, Betreff* — visual asterisk only
How to fix
  • Add required to all mandatory <input> and <textarea> elements.

No Inline Form Validation or Error Messages
MediumAccessibility

No evidence of accessible error messaging on the contact form. On invalid submission, users — especially screen reader users — have no way to identify which fields have errors and why.

No aria-invalid, aria-describedby, or role="alert" elements in contact form markup
How to fix

Inconsistent Semantic Heading Hierarchy
MediumAccessibility

H2 is used for structural labels like "Slideshow", "Suchformular", "Fußbereich" — not content sections. On news pages, H1 "Aktuelle Meldungen" is immediately followed by a duplicate H2. This breaks the document outline for screen readers.

<h2>Slideshow</h2> · <h2>Suchformular</h2> · <h2>Fußbereich</h2>
How to fix
  • Use <section aria-label="Slideshow"> instead of a heading for structural regions.
  • Remove duplicate H2 "Aktuelle Meldungen" that follows the H1 of the same name.
  • Use the HTML5 Outline browser extension to verify heading structure on every template.
  • Rule: headings should reflect content hierarchy, not CSS styling needs.

User Experience

11 issues
Cookie Banner Dominates First Viewport
CriticalUX
Desktop: banner covers lower half of first viewport

The cookie consent banner appears as a large modal-like overlay covering the lower half of the screen on desktop and virtually all content on mobile. Users can't see the page before being forced to interact with legal consent. This creates a terrible first impression and increases bounce rate.

Both desktop and mobile screenshots confirm the banner blocks the primary content area on page load
How to fix
  • Redesign as a slim bottom bar (~80–100px) that doesn't cover content. Full detail available via a "Einstellungen" expand.
  • Show "Alle akzeptieren" and "Alle ablehnen" with equal visual weight — current design buries the reject option.
  • Remember consent in localStorage — don't re-show on every visit.
  • Use a well-tested CMP library (e.g., Klaro, CookieYes) that handles accessibility and mobile layout correctly.
Hero Carousel with 40+ Slides
CriticalUX

The hero slideshow contains 40+ auto-rotating slides. Research consistently shows carousel CTR beyond slide 1 is under 1%. Content in slides 2–40 is effectively invisible. The carousel pattern itself has low engagement — with 40 items it becomes completely unusable.

ubg_bean_slideshow: 40+ slides confirmed in homepage HTML
How to fix
  • Replace with a featured section: 3 editorial cards (image + title + date + CTA) in a responsive grid. No auto-rotation.
  • If a carousel is required by stakeholders, hard-limit to 3–5 slides, manual controls only, and make it opt-in rotation (click to advance).
  • Surface remaining content in the "Aktuelles" grid below the hero instead.
  • KPI to track: hero CTA click-through rate before/after change — expect 5–10× improvement.

News listing — pagination only, no category tags or filters

The news listing provides only numeric pagination across 9+ pages. Users interested in specific topics (Wirtschaft, Bildung, Kommunalpolitik) have no way to filter — they must paginate through everything.

aktuelle-meldungen: pages 1–9+ via numeric pagination only — no tag cloud, category sidebar, or date filter
How to fix
  • Add taxonomy tags to news articles in Drupal (e.g. Wirtschaft, Bildung, Umwelt, Kommunalpolitik).

  • Add a date range picker (year/month dropdown at minimum).

  • In Drupal 7: use the Views module with exposed filters to implement this without custom code.

Contact page: "info[at]cdu-nrw.de" — no mailto: link

HighUX

The "Partei" submenu has 3+ nesting levels (Partei → Gliederungen → CDU vor Ort / Vereinigungen / Fachausschüsse). This creates cognitive overload and is practically unusable on mobile where deep nested menus break.

Partei → Gliederungen → [CDU vor Ort, Vereinigungen, Landesfachausschüsse] = 3 levels

A fixed floating sidebar with social media icons (Facebook, X, YouTube, LinkedIn) is anchored to the right edge of the viewport. This is a design pattern from circa 2014 — it obscures content on smaller desktops and adds no meaningful engagement value as a fixed element.

Split / Inconsistent Header Design
HighUI/Visual
Header: dark left block + white right block — visually fragmented

The header is visually split: a dark teal/black block on the left containing the site title and orange tagline, and a white block on the right containing the CDU logo. These two halves look like separate components that don't belong together. The orange tagline strip is positioned awkwardly below the site title.

Outdated Visual Design Language (Drupal 7 Era)
HighUI/Visual

The site's visual design reflects Drupal 7 era aesthetics (~2011–2016): flat card borders, basic box shadows, tight spacing, and a cluttered layout. For a party with significant political presence, the site underperforms visually compared to peers like die-gruenen-nrw.de or spd-nrw.de.

CMS: Drupal 7 EOL (Nov 2023). Legacy Glyphicons, dated layout patterns visible in screenshots.
How to fix
  • Plan full redesign + CMS migration to Drupal 10/11, WordPress, or headless (Next.js + CMS API).
  • Adopt modern layout: generous white space, large typography, full-width imagery, card-based news grid.
  • Define a proper design system: 8px spacing grid, consistent border-radius (8px standard), one icon library (e.g. Lucide or Heroicons).
  • Benchmark against cdu.de (national) and modern political sites for design direction.
Competing Icon Frameworks Loaded Together
HighUI/Visual

FontAwesome, Material Icons, and legacy Glyphicons are all loaded simultaneously. This creates visual inconsistency (icons from different design languages mixed) and adds unnecessary CSS payload.

CSS imports: FontAwesome + Material Icons + Glyphicon sets — all in the same page

Text Contrast on Header/Navigation Not Verified
LowUI/Visual
Teal nav bar — white text on teal background needs contrast check

The navigation bar uses white text on a teal/cyan background. The orange tagline bar uses dark text on orange. These combinations need formal contrast ratio verification. The teal color appears to be approximately #009aa6 — white on that background may not meet 4.5:1.

How to fix
  • Use the WebAIM Contrast Checker to test every text/background combination.
  • Minimum ratios: 4.5:1 for normal text, 3:1 for large text (18px+ or 14px+ bold).
  • If the teal navigation fails: darken the teal or use white as the nav background with dark text.
  • Also check: orange tagline text, muted footer text, article metadata dates.

Performance

3 issues
50+ CSS Stylesheets Per Page — Render Blocking
HighPerformance

Every page loads 50+ individual CSS files as separate HTTP requests. This is severe render-blocking — the browser must download, parse, and apply each stylesheet before rendering anything. Time to First Contentful Paint (FCP) will be significantly impacted, especially on mobile.

50+ CSS @import / link tags confirmed on aktuelle-meldungen and impressum pages
How to fix
  • Immediate: Enable Drupal 7's built-in CSS aggregation: Admin → Configuration → Development → Performance → "Aggregate and compress CSS files".
  • This alone will collapse 50 files into 1–3 bundles and dramatically improve FCP.
  • Remove unused CSS: audit with Chrome DevTools Coverage tab — delete CSS for unused Drupal modules.
  • Long-term: Migrate to a modern build pipeline (Vite/Webpack) that produces a single optimized CSS bundle.
  • Inline critical CSS (above-fold styles) and defer non-critical CSS loading.
Legacy Drupal 7 CMS (End of Life Nov 2023)
HighPerformance

Drupal 7 is end-of-life. It no longer receives security patches, lacks HTTP/2 push support, uses legacy jQuery, and cannot leverage modern Web Vitals optimizations. Running EOL software on a high-profile political party website is a security and performance liability.

Drupal 7 detected via module names: ubg_bean_slideshow, ubg_socialmedia, cycle.js carousel
How to fix
  • Plan migration to Drupal 10/11 (or headless CMS + Next.js/Nuxt). Allocate 3–6 months for full migration project.
  • In the interim: apply all available Drupal 7 security patches (via drupal.org extended support until Jan 2025, then commercial support).
  • Migrate to Drupal 10's Olivero or Claro themes which are Core Web Vitals-optimized out of the box.
  • Prioritize migration of public-facing pages — internal party tools can migrate later.
Multiple Competing JavaScript Libraries
MediumPerformance

The site loads jQuery, cycle.js, multiple Drupal ubg_* custom modules, and social embed scripts from Facebook/Twitter/Instagram. This adds to Total Blocking Time (TBT) and can cause JS conflicts.

How to fix
  • Enable Drupal JS aggregation: Admin → Performance → "Aggregate JavaScript files".
  • Replace cycle.js carousel with CSS scroll-snap (zero JS, smooth native scrolling).
  • Lazy-load social embed scripts — only inject the <script> tag when the user activates the embed.
  • Audit with Chrome DevTools → Performance → Main Thread to identify which scripts block longest.

GDPR / Legal

2 issues
Privacy Policy Outdated — Obsolete Legal References
HighGDPR

The Datenschutz page is dated May 2018 and still references the EU-US Privacy Shield framework, which was invalidated by the Schrems II ruling (July 2020) and replaced by the EU-US Data Privacy Framework (July 2023). This is an outdated and potentially non-compliant legal basis for US data transfers.

Privacy policy dated May 2018. Privacy Shield §6, §7 references — legally invalid since 2020.
How to fix
  • Immediate legal review required. Engage a GDPR-qualified lawyer to update the policy.
  • Replace Privacy Shield with: Standard Contractual Clauses (SCCs, 2021 version) or EU-US Data Privacy Framework (DPF) certification for US service providers.
  • Update the list of data processors (Union-Betriebs-GmbH, Matomo, Google/YouTube, Facebook/Meta, Twitter/X) with current processing purposes.
  • Add "Zuletzt aktualisiert: [Datum]" at the top and commit to annual review.
  • Check that YouTube embeds use the privacy-enhanced mode (youtube-nocookie.com).
Newsletter Tracking Pixels — Consent Mechanism Unclear
MediumGDPR

The privacy policy mentions tracking newsletter opens via "Web-Beacons bzw. Tracking-Pixel" with pseudonymization. Under GDPR Art. 6(1)(a), tracking individual email opens requires separate explicit consent — it cannot be bundled with the newsletter subscription consent.

Datenschutz: "Web-Beacons bzw. Tracking-Pixel" in newsletter section — no separate consent mechanism described
How to fix
  • Add a separate opt-in checkbox during newsletter signup: "Ich stimme der Auswertung meines Leseverhaltens zu (optional)"
  • Unsubscribed users should also be able to opt out of tracking only (keep newsletter, no tracking).
  • Update the privacy policy to explicitly name the tracking mechanism, legal basis, and opt-out process.
  • In your email tool (e.g. CleverReach, Mailchimp): disable open/click tracking for users who haven't consented.
